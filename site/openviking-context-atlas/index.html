CHAPTER 01 · ORIENTATION

A long context window is not a memory system.

The window is where the model thinks now. A context system decides what can be known, where it lives, what should enter that window, and what survives afterward.

The point

Context engineering is the parent discipline. Retrieval is one read path. Memory adds a write path and a lifecycle.

An agent loop produces a growing universe of possible context: instructions, tools, repository files, customer facts, message history, intermediate results, and lessons from prior runs. Putting all of it into the next call is not neutral. Attention has a budget, stale details compete with current evidence, and a relevant fact can be present yet effectively invisible.

Anthropic’s useful formulation is to optimize the utility of the tokens used for each inference. That reframes the problem from “How large is the window?” to “What is the smallest, highest-signal working set that makes the desired behavior likely?” Context engineering ↗

Inference surface

Prompt & context window

The temporary material visible to the model for one call: instructions, messages, tool descriptions, and retrieved evidence.

ephemeral · token-bounded
Read mechanism

Retrieval-augmented generation

A retriever selects non-parametric evidence for generation. It answers “what evidence should I fetch?”—not the whole lifecycle.

retrieve → condition → generate
Persistent state

Agent memory

Interaction-derived state that can survive, be recalled, revised, merged, or forgotten. Persistence alone does not make its contents true.

write · update · recall · delete
Control plane

Context database

An address space and lifecycle for heterogeneous context assets, plus deterministic and semantic ways to locate and inspect them.

assets · scope · depth · policy

The systems question is larger than top-k

The original RAG architecture joined a generator to explicit non-parametric memory through retrieval. That remains foundational. OpenViking does not eliminate it; it asks what must exist around it: stable addresses, typed assets, hierarchy, progressive representations, session state, write policy, ownership, and an audit trail. RAG, NeurIPS 2020 ↗

A clean separation helps: retrieval proposes candidates, context assembly decides what representation to load under a budget, and memory management decides what interaction-derived state to preserve or change. One product may perform all three, but they should still be evaluated independently.

A precise advocate claim: OpenViking is interesting because it connects the read, manage, and write loops behind an inspectable address space—not because vectors or summaries are new by themselves.
Invariant

The context window is a temporary projection, not the durable knowledge substrate.

Failure pattern

“Add memory” becomes “append more text,” and later calls receive a noisy historical dump.

Diagnostic

Trace every token to a source, selection rule, lifetime, and update owner.

CHAPTER 02 · DATA MODEL

Give context an address.

Anonymous chunks can be similar. Addressed context can also be browsed, scoped, cited, updated, moved, inspected, and deliberately loaded.

The point

viking:// is not decoration. It gives context a canonical, navigable location.

Four axes, not one taxonomy

Asset type and representation depth are only two axes. Ownership scope and cognitive role answer different questions. Keeping them independent prevents the common mistake of reading L0/L1/L2 as short-, medium-, and long-term memory.

AxisQuestionExamples
Asset semanticsWhat kind of thing is this?Resource, Memory, Skill
Representation depthHow much is loaded?L0 abstract, L1 overview, L2 detail
Scope & lifecycleWho owns it, and for how long?Account-shared resources, current user, peer, shared agent skill, session
Cognitive roleHow will the agent use it?Working, episodic, semantic, procedural

A filesystem is a navigation model, not an ontology

Paths preserve neighborhood, naming, and deterministic location. They are especially useful for just-in-time exploration: an agent can inspect a directory, read its overview, then descend. Moving an asset changes its URI and requires relation and index maintenance; the address is canonical location, not immutable object identity. A tree also cannot naturally express every many-to-many relationship, so links, metadata, or a graph remain useful for cross-cutting semantics.

Likewise, a namespace is not authorization by itself. Current code canonicalizes a user root and constrains visible roots, while storage queries also inject tenant filters. Ownership policy—not the string prefix alone—protects isolation. Viking URI ↗

Invariant

The URI identifies an asset even when only a summary—or nothing—is in the current prompt.

Failure pattern

Teams assume a directory is a complete ontology or that a URI prefix is sufficient access control.

Diagnostic

Inspect type, representation, canonical owner, and authorization as separate fields.

CHAPTER 03 · REPRESENTATION

Do not pay full detail before relevance is known.

OpenViking materializes multiple semantic resolutions so an agent can scan cheaply, navigate with context, and open the source only when precision warrants it.

The point

L0 says “is this neighborhood promising?” L1 says “where inside should I go?” L2 answers from the source.

Simplified teaching model

The official model describes L0 as a short abstract used for quick filtering, L1 as a navigational overview, and L2 as original detail. Directory semantics are generated bottom-up: child information helps describe its parent. This makes navigation metadata useful context in its own right. Context layers ↗

The design resonates with hierarchical retrieval research such as RAPTOR, which also retrieves across multiple abstraction levels. The mechanism is different: RAPTOR builds a semantic tree through clustering and recursive summarization; OpenViking generally works over a resource or directory tree and writes explicit semantic sidecars. Similar intuition does not make them the same system. RAPTOR, ICLR 2024 ↗

Version note: documentation alternates between roughly 1k and roughly 2k tokens when describing L1. Treat these as design targets, not a durable API contract. The artifact’s lab uses its own clearly labeled example counts.

The summary is a map, not evidence of every street

Every abstraction can hide a boundary condition, qualifier, or exact value. The safe pattern is progressive: use L0 to shortlist, L1 to navigate, and L2 to support the final precision-sensitive claim. The wrong pattern is to cite a generated overview as if it were the source.

Invariant

Layers express resolution. The same asset can be encountered at L0, L1, or L2.

Failure pattern

A concise summary looks authoritative and silently drops the one exception that matters.

Diagnostic

Use summaries for routing; open the source for exact claims and irreversible actions.

CHAPTER 04 · INGESTION

Write once. Compile several views.

The ingestion path behaves like a compiler: preserve the source tree first, then derive summaries and search records asynchronously.

The point

OpenViking separates what must not be lost from what can be rebuilt.

RequestAdd resourceURL, file, directory, code, media
StructureParser + TreeBuilderMedia parsers may invoke a VLM
Source truthPersist to AGFSURI tree becomes durable
AsyncSemanticQueueBottom-up L0/L1 generation
AsyncEmbeddingQueueMaterialize index records

ResourceService.add_resource() validates the target and starts processing. Parsers convert the source into a temporary structured tree. Under a destination lock, the resource processor persists files and rewrites references, then hands semantic work to the queue. The semantic processor generates directory views bottom-up; a separate embedding queue writes L0, L1, and L2 search records.

This separation is an operational choice, not only a diagram. Model calls and vector services can be slow or unavailable. Keeping source content durable before derived views complete lets the system recover and reindex. It also creates eventual consistency: accepted is not the same as searchable.

Minimal embedded quickstart · current public API
import openviking as ov

client = ov.OpenViking(path="./openviking-data")
client.initialize()

added = client.add_resource(
    path="https://github.com/volcengine/OpenViking",
    wait=True,  # request-scoped completion barrier
)
if any(stage["error_count"]
       for stage in added["queue_status"].values()):
    raise RuntimeError(added["queue_status"])
hits = client.find("How does hierarchical retrieval work?",
                   target_uri=added["root_uri"])

The storage invariant

OpenViking’s transaction guide says the filesystem is the source of truth and the vector database is a derived index. A missing index entry can be rebuilt; lost source content cannot. That is why the architecture prefers a temporarily missed result over an inconsistent one. Transactions ↗

The index is richer than “one vector per chunk.” Current records carry URI, context type, level, dense and optional sparse data, ownership, timestamps, usage metadata, and optional full text. The code can feed different text to embeddings and lexical retrieval, preserving raw text for BM25 while using summary-oriented input for semantic recall.

Failure boundary: enqueue or setup failures may appear as resource warnings; failures after enqueue appear in request or queue status. In either case, a durable tree may already exist while a derived semantic stage has failed. wait=True is a completion barrier only when the caller also inspects every queue group’s error_count.
Invariant

AGFS content is source truth; summaries and vector records are materialized views.

Failure pattern

The API returns success, then an immediate search misses a resource still moving through queues.

Diagnostic

Observe persistence and semantic readiness separately; wait or poll when freshness is required.

CHAPTER 05 · RETRIEVAL

Retrieval is navigation—when the runtime is equipped to navigate.

The source contains both a fast vector path and a recursive directory path. Which one runs depends on configuration, query type, and whether reranking is available.

The point

find is one direct question. search can plan from session context. The retriever then chooses QUICK or THINKING.

Retrieve · context-aware searchFaithful implementation walkthrough
EntryPlanning inputBest useImportant correction
find()One query plus explicit/default target rootsDirect, stateless lookupStill uses HierarchicalRetriever; the runtime mode decides flat versus recursive behavior.
search()Can include archive overview, current messages, and target abstractConversational or ambiguous lookupSession material is optional. Without it, search falls back to a direct typed query.

What THINKING actually does

  1. Embed the query once and resolve policy-approved target roots.
  2. Globally search L0/L1 records for promising starting points.
  3. Rerank directory abstracts when a reranker is configured.
  4. Use a priority queue to expand immediate children in bounded parallel batches.
  5. Blend child relevance with the parent score, deduplicate by URI, and recurse only through directory-level results.
  6. Stop after repeated top-k convergence or search stagnation, then return typed matches.

This is not the only path. QUICK performs one tenant-scoped vector search. The retriever defaults to QUICK when no reranker is available, THINKING when one is, and QUICK/L2 for image queries.

openviking/retrieve/hierarchical_retriever.py · mode selection

Retrieval is still not assembled context

A matched context can carry URI, type, level, abstract, score, and other metadata. The integration still decides whether to use the abstract, open an overview, read full content, cite it, and how much budget to allocate. A high score is an invitation to inspect—not permission to overfill the prompt.

Source correction: the current retriever converts matched contexts with an empty relations list. Documentation that suggests relation expansion in the main retrieval result is ahead of this reviewed implementation path.
Invariant

Recursive traversal is the configured THINKING path, not an unconditional property of every query.

Failure pattern

A team believes hierarchy is active while its deployment is running flat QUICK retrieval.

Diagnostic

Capture mode, planner, targets, reranker, traversed branches, scores, and loaded URIs in one trace.

CHAPTER 06 · SESSION MEMORY

A transcript becomes memory only after policy.

OpenViking’s commit path is not “save the chat.” It snapshots evidence, derives summaries, consults existing memory, and applies structured changes to durable state.

The point

Memory is a compiler output. The session archive is its provenance.

LiveMessages & toolsCurrent interaction evidence
Inline · lockedArchive firstSplit retained tail, persist history
BackgroundSummarizeGenerate archive semantic views
BackgroundExtract & resolveRead/search existing memory
AuditPatch + diffWrite changes, index, mark done

Phase 1 holds an exact session-path lock. It snapshots the messages to archive, keeps the configured recent tail live, creates a tracked task, and returns. Phase 2 runs in the background: hydrate large tool results, generate archive summaries, extract long-term and execution-derived memory, apply structured operations, write memory_diff.json, and place the completion marker last.

The durable-first boundary matters. A crash after archival does not erase the evidence. A pending redo record can retry long-term-memory extraction after an abrupt interruption; it is not a full replay of Phase 2. The phase is not atomic: summary, memory, and execution tasks can succeed or fail independently, and successful sibling outputs are not rolled back.

openviking/session/session.py · the two-phase contract
"""Async commit session: archive immediately,
extract memories in background.

Use is feedback, not truth

session.used() records which URIs helped the current run. After successful Phase 2 processing, OpenViking links the session to those assets and increments their active_count. THINKING retrieval may blend that hotness signal with semantic relevance when hotness_alpha is enabled; QUICK does not. This closes a useful “retrieve → use → remember → retrieve” loop—but popularity remains a ranking signal, never evidence that the content is correct.

Compression, consolidation, and reflection are different

  • Compression reduces representation size while trying to preserve important state.
  • Consolidation turns many observations into more stable summaries, entity state, or patterns.
  • Reflection generates a higher-level lesson or inference from experience.

OpenViking’s commit path performs application-layer consolidation. “Self-evolving” means external state changes and conditions later inference. It does not mean the base model’s weights changed, the lesson is verified truth, or the core Session will auto-commit at a token threshold. In this snapshot, integrations trigger commit explicitly.

Invariant

Raw evidence is archived before background memory compilation can fail.

Failure pattern

Task acceptance is shown as “memory saved,” even though extraction remains pending or partial.

Diagnostic

Poll task state; inspect archive, completion marker, and memory diff; test the next recall.

The filesystem is the interface. The runtime is a service graph.

At the center, OpenVikingService wires business services to storage, queues, models, processors, identity, and operational controls.

The point

The architecture earns depth by hiding machinery behind a small vocabulary: add, browse, find, search, read, session, commit.

The composition root first initializes configuration, process and path locks, AGFS bindings, queues, the vector manager, and an embedder. During asynchronous initialization it creates the shared context collection, builds VikingFS with retrieval and rerank configuration, initializes canonical directories, then attaches resource, skill, memory, privacy, watch, and debug components.

Business services receive the same underlying VikingFS and processors. That is why embedded and server deployments can expose similar semantics instead of duplicating logic in each transport. The service layer is not merely architectural ceremony: it is the seam between public API behavior and the stateful runtime.

A useful “context operating system” analogy—with limits

The system has symbolic addresses, multiple representations, policy-scoped visibility, background compilation, and paging-like progressive reads. That makes “context operating system” a productive mental model. It is still an application service, not a kernel, a literal disk filesystem, or a replacement for the agent harness that assembles each model call.

Invariant

Public transports converge on a shared service and engine graph.

Failure pattern

The filesystem metaphor hides operational dependencies and eventual-consistency boundaries.

Diagnostic

Trace one call through transport, service, engine, storage, index, and queued work.

CHAPTER 08 · OPERATIONS

The database earns its name at the failure boundaries.

Production quality is not “the server starts.” It is knowing what survives, what can lag, what can be rebuilt, who can see it, and how a bad memory is found and removed.

The point

Source durability, semantic freshness, memory correctness, and authorization are four different promises.

Consistency

Path locks + redo

Core writes acquire path locks. A pending redo record can retry long-term-memory extraction after interruption; it is not full Phase 2 replay.

protect source · retry extraction
Freshness

Queues + task state

Semantic and embedding work is decoupled. A caller needs explicit readiness when immediate recall is part of the contract.

accepted ≠ indexed
Isolation

Identity + visible roots

Canonical user and peer scopes combine with access checks and tenant filters. Namespaces assist policy; they do not replace it.

owner · actor peer · role
Governance

Diffs + deletion + reindex

Durable state should be inspectable, correctable, and removable. A derived index must follow source changes or be rebuilt.

audit · repair · forget

Six concrete risks

  1. Eventual consistency: source content exists before summaries and vectors are ready.
  2. Model-dependent materialization: L0/L1 quality and memory extraction inherit model errors, cost, latency, and drift.
  3. Memory pollution: a plausible extracted lesson can be false, overgeneralized, scoped to the wrong peer, or stale.
  4. Partial background success: commit Phase 2 is recoverable but not transactionally all-or-nothing.
  5. Context injection: stored resources and memories can contain instructions that should never outrank trusted policy.
  6. Version drift: docs, schemas, tools, and defaults can diverge; production behavior must be pinned and observed.
Security boundary: a context database can make untrusted text easier to retrieve. It cannot decide that retrieved instructions are safe to follow. Preserve provenance, separate data from policy, enforce instruction hierarchy in the harness, and test poisoned-memory scenarios.

CHAPTER 09 · INTELLECTUAL LINEAGE

OpenViking belongs to a field, not a vacuum.

Its strongest case is a synthesis: retrieval, progressive abstraction, virtual context, reflection, durable state, filesystem navigation, and database operations brought into one inspectable control plane.

The point

Credit the lineage precisely. The new value is the shape of the whole loop.

2020

RAG

Couples generation to explicit non-parametric memory through retrieval, foregrounding updateability and provenance.

2023

Generative Agents · CoALA · MemGPT · Reflexion

Memory streams and reflection, cognitive memory/action taxonomies, virtual context tiers, and textual feedback across trials.

2024

RAPTOR · GraphRAG

Develops two different structural moves: retrieval across recursively summarized abstractions, and graph/community views for local-to-global questions.

2025

Context engineering

Frames every inference as selection under a finite attention budget and promotes just-in-time exploration through stable identifiers.

2026

The cognitive analogy is useful if it stays an analogy

CoALA’s working, episodic, semantic, and procedural vocabulary helps describe how an agent may use stored material. OpenViking’s live session resembles working evidence; events, cases, and trajectories can serve episodic roles; entities and preferences can serve semantic roles; tools and skills can serve procedural roles. These are explanatory mappings across independent taxonomies—not biological claims. CoALA ↗

MemGPT’s operating-system analogy explains why limited model context benefits from external tiers and explicit movement. Anthropic’s context-engineering guidance explains why lightweight paths and progressive exploration are valuable. OpenViking’s contribution is to give those instincts a database-shaped, inspectable substrate. MemGPT ↗

Paper boundary: the OpenViking repository says it open-sources a subset of the VikingMem paper’s capabilities. Do not automatically attribute the paper’s event/entity evolution or temporal weighting mechanisms to every OpenViking code path. VikingMem ↗

Why this synthesis is worth advocating

An agent can keep a URI instead of a whole document, inspect a neighborhood before opening a file, retrieve semantically without surrendering deterministic navigation, preserve raw interaction evidence before extracting a lesson, audit what memory changed, and rebuild search views from source truth. None of those moves is magic. Together they form a practical answer to the central context-engineering problem: put the right evidence, at the right depth, into the right call—and preserve a trail back out.

Invariant

The value is the composed lifecycle, not a claim to have invented every component technique.

Failure pattern

Neighboring labels are used as synonyms, making comparisons impressive but technically false.

Diagnostic

Compare unit, hierarchy, read policy, write policy, provenance, and failure mode.

CHAPTER 10 · FIELD PRACTICE

Build a context policy, not just a connector.

The integration is finished when the team can explain what is stored, who sees it, how it enters the prompt, when it changes, and how failures are detected.

The point

Start with one end-to-end trace. Automate recall and commit only after the boundaries are observable.

Embedded end-to-end skeleton · adapted from an integration test
import asyncio
from openviking import AsyncOpenViking
from openviking.message import TextPart

client = AsyncOpenViking(path="./openviking-data")
await client.initialize()
added = await client.add_resource("./handbook.md", wait=True)
if any(stage["error_count"]
       for stage in added["queue_status"].values()):
    raise RuntimeError(added["queue_status"])

session = client.session(session_id="field-guide-demo")
session.add_message("user", [TextPart("How should this run?")])
hits = await client.search("deployment policy", session=session)
if hits.resources:
    session.used(contexts=[hits.resources[0].uri])

session.add_message("assistant", [TextPart("…answer…")])
commit = session.commit()  # accepted result contains task_id
while True:
    task = await client.get_task(commit["task_id"])
    if task and task["status"] in {"completed", "failed"}:
        break
    await asyncio.sleep(0.2)
if task["status"] != "completed":
    raise RuntimeError(task)

The happy path is only half the design

Also exercise one summary failure, one delayed index, one empty retrieval, one conflicting preference, one wrong-peer query, one commit interrupted before .done, and one deletion followed by reindex. A context system becomes trustworthy through these failure traces, not through a single fluent answer.

The smallest useful next action

Pick one real recurring agent task. Ingest only its authoritative resources. Run one search with telemetry, load the minimum useful depth, record the used URIs, commit one session, inspect the memory diff, then repeat the task. If the second run improves for the right reason—and unrelated tasks do not regress—you have the beginning of a memory system rather than a memory demo.
